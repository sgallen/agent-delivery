# Playbook Map

This playbook has four layers. Read only as deeply as the problem requires.

## 1. Understand the shift

Start with [Why This Exists](00-the-case/00-why-this-exists.md), then read [The Case](00-the-case/index.md).

It explains why current agents are more than a productivity feature, why builder attention becomes the bottleneck, and why the operating model needs to change.

Recommended short path:

1. [Why This Exists](00-the-case/00-why-this-exists.md)
2. [The Economics Changed](00-the-case/01-the-economics-changed.md)
3. [The Cost Curve Made Concrete](00-the-case/02-the-cost-curve-made-concrete.md)
4. [Invert the Default](00-the-case/06-invert-the-default.md)
5. [The North Star](00-the-case/10-the-north-star.md)

## 2. Understand the system

Read the [Operating Model](01-operating-model/index.md).

Start with:

1. [Changes](01-operating-model/changes.md)
2. [`WORKFLOW.md` and Workpads](01-operating-model/workflow-and-workpads.md)
3. [Gates](01-operating-model/gates.md)
4. [Evidence](01-operating-model/evidence.md)
5. [Builder Engagement](01-operating-model/builder-engagement.md)

## 3. Try it

Use the [First Experiment](02-playbooks/first-experiment.md) to run one low-risk Change through the model. For a concrete bug workflow, use [Fix a Bug with Proof](02-playbooks/fix-a-bug-with-proof.md).

For a team-level start, use [The First 30 Days](02-playbooks/first-30-days.md), then choose a path from the [Playbooks](02-playbooks/index.md): greenfield, existing-team adoption, GitHub workflow, gates, browser proof, environments, economics, or recovery.

## 4. Implement it

Use the [Reference](03-reference/index.md) and `templates/` directory for copyable starting points.

The templates demonstrate one GitHub-first implementation. Keep the operating principles; adapt the tools to the project.
