# Operating Model

The Case explains why the shift matters. The operating model explains how to respond without turning the repository into a haunted house of agent-generated branches.

The aim is not more process. It is enough structure for agents to do useful work, builders to apply judgment where it matters, and the team to learn from what delivery actually consumed and produced.

Manage Changes and outcomes, not interactive sessions. Let every artifact earn its place by improving intent, execution, proof, recovery, or a decision.

## The system at a glance

```text
Initiative, when the work is a meaningful product bet
  → value hypothesis and resource range

Change Intent
  + repo-level WORKFLOW.md
  → one persistent workpad
  → isolated execution and attributable runs
  → gates, evidence, and resource checkpoints
  → builder decision
  → delivered work, a useful non-landed decision,
    accountable closure, or visible unresolved loss
  → learning and, when relevant, later value review
```

## Core concepts

- [**Builders**](builders.md) own outcomes, judgment, taste, risk, value, and the system itself.
- [**Agents and harnesses**](agents-harnesses-and-operating-model.md) provide the execution capability and the technical conditions around it.
- [**Changes**](changes.md) carry one piece of work from intent through evidence, resolution, and learning.
- [**`WORKFLOW.md` and workpads**](workflow-and-workpads.md) separate the standing repository contract from the live execution state.
- [**Gates**](gates.md) make quality, behavior, risk, and resource policy durable.
- [**Evidence**](evidence.md) makes the outcome easier to judge.
- [**Builder engagement**](builder-engagement.md) places scarce human attention where it changes the result.
- [**Resource observability and delivery economics**](resource-observability-and-delivery-economics.md) connect machine use, infrastructure, time, and builder attention to outcomes.
- [**Initiatives and the value loop**](initiatives-and-value.md) connect product bets to forecast investment, actual delivery, and realized value.
- [**Learning checkpoints**](learning-checkpoints.md) turn repeated friction and forecast variance into a better operating system.

## The smallest credible version

A first implementation needs only:

1. a clear Change Intent;
2. a short `WORKFLOW.md`;
3. one persistent workpad;
4. an isolated enough environment;
5. a few meaningful gates;
6. evidence tied to the intended outcome or decision;
7. lightweight attribution of runs, builder attention, and final resolution;
8. one honest learning checkpoint.

Do not wait for a forecasting model, a hard budget policy, or a portfolio dashboard. Those are later moves. Earn them with credible Change records first.
