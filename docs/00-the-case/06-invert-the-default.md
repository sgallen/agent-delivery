# Invert the Default

A common objection is straightforward:

> Agents cannot do all the work. They will produce slop, and the slop will be impossible to review.

That concern is real. It is also more useful as an engineering problem than as a reason to preserve the old process.

The limiting question is:

> Where can we add agents to help the existing workflow?

That assumes the human-led workflow remains the center of gravity.

Try a harder thought experiment:

> If the agent-enabled system attempted the work by default, where would it fail—and where would human judgment create the most leverage?

Now put on the engineering hat.

What context is missing? What tools are required? What boundaries must be enforced? What evidence would make the outcome easy to judge? Which risks require product, design, engineering, security, or business judgment? Which repeated failures should become tests, skills, gates, or workflow improvements?

The point is not to pretend agents are perfect or force every Change through the same path.

The point is to design around agent execution, then engage builders deliberately:

- to shape intent;
- to resolve ambiguity;
- to apply taste;
- to make high-consequence decisions;
- to judge evidence;
- to improve the system.

The goal is not maximum autonomy.

The goal is maximum leverage from builder attention.
