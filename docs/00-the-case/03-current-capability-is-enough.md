# Current Capability Is Enough

The case for agent-enabled product delivery does not depend on future model improvement.

Current agents are already capable of meaningful product development work. They can use tools, inspect feedback, repair failures, produce evidence, and participate in the path from intent to shipped product change.

That is enough to justify a different operating model.

## This is no longer hypothetical

OpenAI's [Harness Engineering](https://openai.com/index/harness-engineering/) team describes shipping a real internal product with all repository code written by Codex, estimating roughly one-tenth the hand-written development time and reporting about 1,500 merged pull requests during the first five months.

The [Gusto Cofounder story](https://youtu.be/5FKBkUCaLa8) describes a five-person team taking a new product line from zero code to a major launch in 10 weeks with Claude Code at the center of the build process.

Those are not universal productivity benchmarks, and copying their surface process would be a poor substitute for judgment. They are concrete evidence that the capability has moved beyond snippets and demos.

A team can add AI tools to the existing workflow and get faster at the existing workflow. It can draft specs faster, generate code faster, summarize meetings faster, and open pull requests faster.

Useful? Certainly.

The full advantage comes from redesigning the system itself:

- how work is shaped
- how agents receive context and tools
- how quality is expressed through gates
- how evidence is produced
- how review bottlenecks are reduced
- how useful learning becomes durable.

Even if models stopped improving tomorrow, a team that built that system would have an advantage over a team that merely accelerated old handoffs.

Future capability is upside, not a dependency.
