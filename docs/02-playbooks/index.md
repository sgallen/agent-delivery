# Playbooks

Playbooks turn the doctrine into action.

Use them when the idea is no longer interesting in the abstract and you need to make it work. This is where the useful learning starts.

- [Resolve a Change that does not land](resolve-a-change-that-does-not-land.md) — turn an experiment, stop, rejection, or superseded path into an evidence-backed resolution without disguising abandonment as learning.

## How to use these

Do not adopt every playbook at once.

Start with one low-risk project or one safe class of work. Build the smallest version of the system that can produce trustworthy evidence, attribute the run to a Change, and record an explicit disposition whether the work lands or not. Then improve it from what the run teaches you.

Initiative forecasting, hard resource gates, learned estimators, and portfolio capacity planning are later maturity moves. They become useful when the underlying Change records are consistent enough to support a real decision.

The operating model should grow from reality, not from an afternoon of enthusiastic diagramming.
