# Playbooks

The doctrine becomes useful when someone can apply it on Tuesday morning.

Start with one real Change. Add the more advanced practices only when the work gives you a reason.

## Begin here

- [**Run the First Experiment**](first-experiment.md) — test the smallest credible operating model on one low-risk Change.
- [**Fix a Bug with Proof**](fix-a-bug-with-proof.md) — turn a real defect into a report → reproduce → fix → prove → learn loop.
- [**The First 30 Days**](first-30-days.md) — establish a team-level path, run a small cohort, and decide what has earned expansion.

## Learn the economics

- [Measure Delivery Economics](measure-delivery-economics.md) — connect resource use to delivered work, decisions, and unresolved loss.
- [Resolve a Change That Does Not Land](resolve-a-change-that-does-not-land.md) — stop, reject, or redirect work without pretending every dead branch was valuable.
- [Forecast and Review an Initiative](forecast-and-review-an-initiative.md) — connect a product thesis to forecast investment, actual delivery, and later value.
- [Plan Delivery Capacity](plan-delivery-capacity.md) — find the human or technical constraint that determines how much agent capacity the team can usefully absorb.

## Adopt without creating a new religion

- [Start a Greenfield Project](start-greenfield.md)
- [Shift an Existing Team](shift-existing-team.md)

Do not adopt every playbook at once. Begin with the smallest path that can produce trustworthy evidence and an explicit outcome. Expand because reality earned the next layer, not because the diagram looked reassuring.
