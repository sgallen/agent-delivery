# Agent Instructions

This repository is a Markdown-first playbook for human-governed, agent-enabled product delivery.

The job is not to add more doctrine. It is to make the shift easier to understand, easier to challenge, and easier to try.

## Read before editing

Start with:

1. `README.md`
2. `VOICE.md`
3. `docs/00-the-case/00-why-this-exists.md`
4. `docs/01-operating-model/index.md`
5. the pages, references, and templates directly touched by the Change

For economics, initiative, or non-landed-work edits, also read the relevant source pages:

- `docs/01-operating-model/resource-observability-and-delivery-economics.md`
- `docs/01-operating-model/initiatives-and-value.md`
- `docs/01-operating-model/not-all-work-is-merge-bound.md`

## Preserve the hierarchy of the argument

The centre of gravity is:

1. Current agents can perform meaningful product-development work.
2. Traditional delivery was shaped around scarcer human implementation capacity.
3. The stronger response is to redesign the operating model, not merely accelerate the old one.
4. Builders own intent, judgment, risk, taste, and outcomes. Agents execute inside a legible, bounded, provable system.
5. Changes, `WORKFLOW.md`, workpads, environments, gates, evidence, recovery, and learning make that system practical.
6. Advanced practices—resource attribution, estimates, thresholds, routing, capacity planning, initiative economics, and value review—become possible as trustworthy records accumulate.

Do not let the Change become the thesis. It is the practical unit that carries the thesis into daily work.

Do not let economics become the thesis either. It is a valuable learning and governance layer that a mature operating model unlocks.

Not every planned Change must land. A useful non-landed decision needs evidence. Administrative closure and unresolved loss remain distinct when the formal vocabulary matters.

Do not imply that quality matters less, that specialists are obsolete, or that agent spend and human capability are interchangeable forms of capacity.

## Write in the repo’s voice

Follow `VOICE.md`.

Lead with the consequence the reader can feel. Explain the idea before the taxonomy. Prefer short paragraphs, concrete verbs, and one main job per page. Use lists when they help someone scan or act—not when prose would carry the thought better.

Keep the edge in the clarity. Avoid vendor language, process theater, academic distance, and a machine explaining its own ontology.

The reader should leave with a sharper way to see the problem and a practical next move.

## Keep the package coherent

- Let the README invite, orient, and carry the full high-level argument.
- Let The Case persuade in sequence.
- Let the operating model explain how the system works.
- Let playbooks tell someone what to do.
- Let reference pages and templates stay terse, exact, and copyable.
- Keep advanced practices clearly progressive. Do not make the first credible Change complete an enterprise data model.
- Keep taxonomy in its natural home. Link rather than re-teaching it on every page.
- Keep templates useful on Tuesday morning. Optional fields should remain optional unless the consequence makes them necessary.
- Preserve the distinction among Run, Change, Initiative, and Portfolio.
- Use ranges, confidence, assumptions, and data-quality notes. Do not manufacture precision from sparse history.
- Keep volatile model prices dated, sourced, and consistent with `data/model-pricing.csv`.
- Keep human time separate from any optional monetary conversion. Prefer team and capability patterns over individual surveillance.

## Before resolving an edit

Check that:

- the core argument is clearer or the next action is easier;
- the newest concept did not crowd out the larger thesis;
- repetition fell rather than moved;
- links and artifact paths resolve;
- terms and lifecycle states remain consistent;
- templates still match the doctrine;
- code fences, YAML, JSON, CSV, shell, and Python still validate where touched;
- the Change did not add structure that fails to improve intent, execution, proof, recovery, judgment, or learning.

The standing test is simple: a serious builder should be able to read the page, recognize the problem, and know what to try next.
